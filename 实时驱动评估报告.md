# HeyGem ONNX 实时摄像头驱动 — 可行性评估报告

**日期:** 2026-05-11  
**目标:** 摄像头画面 → HeyGem 对口型 → OBS 虚拟摄像头 → 直播平台

---

## 1. 当前架构分析

### 1.1 管线流程

```
音频文件(.wav)
    │
    ▼
wenet ASR 特征提取 ──────────┐
                              │
视频文件(.mp4)                ├──► DINet 人脸渲染 ──► GFPGAN 超分 ──► ffmpeg 编码 ──► 输出.mp4
    │                         │
    ▼                         │
SCRFD 人脸检测 ──► 3DMM 关键点 ┘
```

### 1.2 技术栈

| 组件 | 实现方式 | 可控性 |
|------|---------|--------|
| 完整管线 | `trans_dh_service.cpython-39.so` | 不可控 — C 扩展黑盒 |
| 人脸检测 | `scrfd.cpython-39.so` | 不可控 |
| 人脸对齐 | `face_align_5_landmarks.so` | 不可控 |
| 超分辨率 | `gfpgan_onnx_api.so` | 不可控 |
| 模型推理 | `onnxruntime.InferenceSession` | **可控 — 标准 ONNX API** |
| ONNX 模型文件 | `pretrain_models/**/*.onnx` | **可控 — 本地文件** |

### 1.3 已测性能数据

| 指标 | 数值 |
|------|------|
| 30秒视频总处理时间 | 221 秒 |
| 慢于实时倍数 | **7.4x** |
| DINet 渲染单帧 | ~0.02s (50fps) |
| GFPGAN 超分单帧 | ~0.38s (2.6fps) |
| GFPGAN 占总时间 | **~80%** |

### 1.4 `.so` 黑盒的限制

- 内部硬编码批处理架构，不支持流式输入
- GFPGAN 开关/参数无法外部配置
- 队列逻辑不可修改（已触发过溢出崩溃）
- 仅支持 Linux，无法在 Windows 原生运行

## 2. 实时化方案评估

### 2.1 方案对比

| 方案 | 做法 | 延迟 | 帧率 | 画质 | 工作量 |
|------|------|------|------|------|--------|
| A. 纯 Python 重写管线（推荐） | 绕过 .so，用 ONNX 模型直接推理 | 1-3秒 | 15-25fps | 中等（无GFPGAN） | 2-3周 |
| B. .so + 切片 | 在现有 .so 上做滑动窗口切片 | 3-5秒 | 8-12fps | 高（有GFPGAN） | 1周 |
| C. 升级硬件 | 换更高端 GPU | 改善有限 | 改善有限 | 不变 | 0（花钱） |

### 2.2 推荐方案 A：纯 Python + ONNX Runtime

**架构：**

```
摄像头(Windows) ──► Python 帧捕获
                         │
                         ├──► SCRFD 人脸检测 (ONNX, ~10ms)
                         ├──► 3DMM 关键点 (ONNX, ~2ms)
                         ├──► DINet 渲染 (ONNX, ~20ms)
                         │
麦克风(Windows) ──► Python 音频捕获
                         │
                         ├──► MFCC/Wenet 特征 (~5ms)
                         │
                         ▼
                    合成帧 ──► pyvirtualcam ──► OBS ──► 直播平台
```

**关键优势：**

- 每步可独立优化，瓶颈精准定位
- 去掉 GFPGAN 只需删一行代码
- 支持 FP16 半精度推理（1.5-2x 加速潜力）
- Windows 原生运行，天然支持虚拟摄像头
- 可以做真正的流式滑动窗口，不用切片

**理论性能：**

| 阶段 | 单帧耗时 |
|------|---------|
| 人脸检测 (SCRFD) | 5-10ms |
| 关键点提取 (3DMM) | 1-3ms |
| 音频特征 (MFCC) | 2-5ms |
| DINet 渲染 | 15-25ms |
| **合计** | **23-43ms (23-43fps)** |
| 加上 Python/内存开销 | ~50-65ms (15-20fps) |

## 3. GPU 显存分析

### 3.1 当前显存占用估算

| 模型 | 占用 |
|------|------|
| SCRFD 人脸检测 | ~200MB |
| 人脸解析 | ~500MB |
| DINet 渲染网络 | ~1-2GB |
| GFPGAN 超分 | ~2-3GB |
| 音频特征 + 帧缓冲 | ~1GB |
| **总计** | **约 5-7GB** |
| **去掉 GFPGAN 后** | **约 3-4GB** |

### 3.2 结论

- 当前 RTX 50 系（≥8GB VRAM）**已完全够用**
- 更大显存不会提升推理速度
- 真正的瓶颈是 **CUDA 核心算力**，不是显存容量
- 如果跑多路并发（多直播间），才需要更大显存

### 3.3 真正有效的提速手段

| 手段 | 预期加速 | 代价 |
|------|---------|------|
| 去掉 GFPGAN | **20x** | 人脸清晰度下降 |
| FP16 半精度推理 | 1.5-2x | 精度损失可忽略 |
| TensorRT 引擎 | 2-3x | 需重新导出模型 |
| CUDA Graph | 1.2-1.3x | 实现复杂度高 |
| 换 RTX 4090 | ~1.5x | 金钱成本 |

## 4. 风险与挑战

| 风险 | 等级 | 说明 |
|------|------|------|
| DINet 模型输入格式未知 | 中 | 需要逆向分析 .so 或查论文确定张量 shape |
| 部分逻辑在 .so 中无文档 | 中 | 3DMM 参数提取、人脸 warping 的具体实现需逆向 |
| 音频特征方案选择 | 低 | MFCC 简单但口型精度不如 wenet；wenet 需额外集成 |
| 虚拟摄像头兼容性 | 低 | pyvirtualcam 在 Windows 上成熟可用 |
| WSL → Windows 迁移 | 低 | ONNX Runtime CUDA 在 Windows 上完全一致 |
